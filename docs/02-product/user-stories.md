## 核心故事（P1 范围）

### US-1 从想法到需求文档
**作为**王校长，**我想**用大白话描述"我要一个课时管理工具"，**以便**不写文档也能把需求说清楚。

- AI 追问使用者/现状/时间线，右侧需求草稿实时更新
- 给出 SaaS/低代码/自研三方案对比；**如果现成产品更合适，允许我选"买现成"结束**（诚实原则）
- 我看得懂每个字段，能直接说"改一下 XX"
- 信息不足时"拍板"按钮不出现（双条件门禁）

### US-2 确认需求与能力边界
**作为**王校长，**我想**在开工前确认清单式的需求文档，**以便**知道做出来的东西按什么标准算合格。

- 验收标准由我确认，之后作为验收清单逐条出现
- 若我的需求涉及第三方对接（微信/短信），系统**明确告知这部分是演示**，我确认后才继续

### US-3 选型与风格
**作为**李工，**我想**在网页/小程序/桌面之间得到一句人话的建议，**以便**不懂技术也能选对形态。

- 每个选项标注"适合你吗 + 代价"；AI 有推荐标记
- 风格 A/B 可预览，可以说"颜色暖一点"

### US-4 制作与试用验收
**作为**李工，**我想**几分钟内拿到能实际操作的应用并按我的验收标准逐条勾选，**以便**确信它真的能用。

- 制作过程有进度反馈；完成后试用预览可真实录入数据
- 录入进度 65% 的节点自动标红预警（我的业务规则生效）
- 验收清单 = 我在 US-2 确认的标准，不是系统写死的
- 勾不满可点"还不行，继续改"，我发的下一条消息就是修改要求；改完原数据和其他功能不受影响

### US-5 上线与真实使用
**作为**王校长，**我想**把应用交给我的 3 个教务老师每天用，**以便**替换掉 Excel。

- 测试环境先给同事试，上线检查清单确认后发布正式环境
- 每个老师有自己的账号；**换电脑、隔天登录，数据都在**
- 可导出 CSV 对账

### US-6 无密钥演示（获客场景）
**作为**平台运营，**我想**在任何环境（含静态托管）零配置演示全流程，**以便**给潜在客户看效果而不消耗额度。

- "先用体验模式"一键进入；顶栏明确标注"体验模式（脚本演示）"

## 用例：异常路径（P1 必须覆盖）

| 用例 | 期望行为 | 状态 |
|---|---|---|
| 制作失败（LLM 错误/超时） | 失败手册出现：重试制作 / 改需求 / 用 mock 预览继续；错误详情可展开 | ✅ 已实现 |
| 引擎未连接发消息 | 输入禁用 + 连接引导弹窗 | ✅ 已实现 |
| 静态托管环境（无后端） | 探测失败 → 提示"当前为静态托管，建议体验模式" | ✅ 已实现 |
| 数据底座启动失败 | 制作引导返回明确错误（"数据底座启动失败：…"） | ✅ 已实现 |
| 需求超出模板引擎能力 | 教练如实告知边界，引导拆分或人工介入 | ⚠️ 依赖提示词，稳定性实验观察 |
