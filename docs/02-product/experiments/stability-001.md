# 稳定性实验 001：3 需求 × 2 次 + 修改请求

> 状态：**已完成（2026-07-05）** ｜ 协议见 [验收标准与测试场景 §3](../acceptance-test-scenarios.md#3-稳定性实验协议p1-退出门禁)

## 实验环境

| 项 | 值 |
|---|---|
| 引擎 / 模型 | OpenHands agent-server + DeepSeek（云端注入密钥 `stagent`） |

## 汇总与判定

| 指标 | 结果 | 阈值 | 判定 |
|---|---|---|---|
| 首次生成成功率 | **6/6 = 100%** | ≥80% | ✅ 通过 |
| 修改请求成功率 | 4/6 = 67%（按实际执行 4/5 = 80%） | ≥70% | ⚠️ 边界（见分析） |
| 全流程上线成功率 | 3/6 = 50% | —（未设阈值） | 主因已修复 |
| 平均制作耗时 | ~1.5 min | ≤5 min | ✅ 通过 |
| `c_create`（全流程） | **~$0.02/次 ≈ ¥0.14** | 实测回填 | ✅ 已回填 |

**分析**：
- 修改失败的两例：B1 是修改轮 token 爆炸（155 万 prompt，全文件重写模式的固有风险——baas 档位的声明式 modules.js 已结构性缓解，另加提示词强化）；C1 是测试流程未执行（非产品失败）。按实际执行口径 4/5=80% 达标，按严格口径 67% 差 1 例。
- 上线失败 3 例中 2 例（B1 门禁、B2 会话丢失）根因是 **P0 门禁状态机 bug**（coach 重复发送未变化的 needs 字段导致 feasibilityAcknowledged 被重置）与测试会话操作问题，**非生成能力问题**。

## 问题清单与处置

| 级别 | 问题 | 处置 |
|---|---|---|
| P0 | coach 重复携带未变化的 needs 字段时 feasibilityAcknowledged 被重置，门禁不出现（B1 卡 87min） | ✅ 已修复（store.ts 仅在值实际变化时重置）+ 回归场景 R14 |
| P1 | coach 定稿时省略 feasibilityAcknowledged:true | ✅ 提示词已强化（必须持续携带） |
| P1 | 修改轮偶发 token 爆炸 | ✅ 提示词强化（增量修改、两次编辑内完成）；baas 档位结构性缓解；持续观察 |
| P2 | coach 说「右侧确认」与实际按钮在左侧不符 | ✅ 提示词增加引导文案规范 |
| P2 | 自动化测试会话丢失导致验收未走完 | 流程问题，交付验收单以人工操作为准 |
| P3 | 重试后聊天保留历史错误 | 按约定不修复 |

## 结论与后续

- **P1 内测放行判定：有条件通过**——首次生成 100%、成本与耗时优秀；修改成功率按执行口径达标；门禁 P0 已修复。
- 条件：① 对 B1 场景补跑 1 次「需求确认 → 制作 → 修改」验证门禁修复与 token 行为；② **本实验跑在 static-mvp 档位，baas-mvp（现行 main）需按同协议补测**（追加 R6 持久化断言）——建议作为稳定性实验 002。
- 实验 002 每次生成后用 `npx tsx tests/app-assertions.ts <conversationId> <slug>` 自动验收 R6+R9（免人工操作，规避 001 的操作失误）。
- 单位经济结论已回填 [business-strategy.md](../../01-strategy/business-strategy.md) 与 [metrics.md](../metrics.md)。
