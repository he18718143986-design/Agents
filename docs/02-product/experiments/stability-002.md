在正式 6 轮前，对 B1 场景单独验证 P0 门禁修复（coach 重复携带未变化的 `needs*` 字段后「确认需求，继续」不消失）：

| 项 | 结果 |
|---|---|
| 场景 | 小餐馆进销存，coach 多轮追问后发送「需求没有要补充的了，请定稿」 |
| R14 回归 | `npx tsx prototype/mvp-ui/tests/r14-gate-regression.ts` → **PASS** |
| 浏览器实跑 | 左侧底部红色门禁按钮持续可见，点击后成功进入阶段 2 → **PASS** |
| 备注 | UI 文案可能为「确认概要，继续」，与「确认需求，继续」为同一门禁位 |

## 结论

- **baas-mvp 档位首次生成 83%、修改 80%（执行口径）、R9 全通过**，达到 P1 内测放行阈值。
- 相对 static-mvp，真实持久化链路（PocketBase + 登录 + 体检 CRUD）在 B1/B2/C1/C2 上验证通过；B1 门禁回归不再复现 001 的 87min 卡死。
- **A1 修改轮、A2 首轮**建议各补跑 1 次（不改代码前提下）以闭合严格口径 6/6。
- 单位经济：`c_create` 约 **$0.011/次**，已优于 001 的 ~$0.02/次。
